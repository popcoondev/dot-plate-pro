# Dot Plate Pro 開発ワークフロー

最終更新日: 2026-04-28

## 方針

Dot Plate Pro の開発は、**BDD + チケットドリブン + feature ブランチ駆動** で進めます。

- 運用ルールの正本は `docs/` 配下とする
- バックログとチケット本文の正本は GitHub issue とする
- `docs/` にチケット本文を重複記載しない
- 実装前に必ず GitHub issue を作る

- 1チケット = 1feature ブランチ = 1PR を原則とする
- 1機能1チケットを標準粒度とする
- すべてのチケットに最低 1 本の `Given / When / Then` を入れる
- 技術的負債、依存更新、脆弱性対応、調査もチケット化対象に含める

## 基本ルール

### チケット起点

- 実装は必ず GitHub issue 起点で開始する
- 1つの PR には原則 1チケットのみを紐づける
- 複数の責務を 1つのチケットや PR に混ぜない
- issue template を標準入口とし、BDD、受け入れ条件、テスト観点を GitHub 上で埋める

### ブランチ運用

- ブランチ命名規則は `feature/ticket-{id}-{slug}` を標準とする
- 例: `feature/ticket-12-auto-outline-tool`
- チケット番号未採番の作業は原則開始しない

### 完了条件

- チケット内の BDD シナリオが満たされている
- 受け入れ条件がすべて満たされている
- チケットに定義したテスト観点の確認が終わっている
- 1PR 1責務の状態が保たれている

### 実務上の追加ガード

- ユーザー確認が必要な UI / 操作系チケットは、確認完了まで issue を close しない
- 完了扱いになったチケットは、`main` に取り込み、`origin/main` まで push してから次の作業へ進む
- 実装中は「どのブランチ」「どの worktree」「どの dev server」を見ているかを必ず一致させる
- 別 worktree で作業した場合は、ユーザーが実際に確認している環境へ反映されていることを明示的に確認する
- ユーザーへ動作確認を依頼する前に、ユーザーが見ている作業フォルダに対象差分が入っていることを `rg` や `git diff` で必ず確認する
- 未完了チケットの差分を、別チケットのブランチやコミットに混ぜない
- マージ前に、最後の修正が未コミットで残っていないことを必ず確認する

## チケットの標準粒度

標準粒度は **1機能1チケット** です。

以下はすべてチケット化します。

- 機能追加
- バグ修正
- UI 改善
- リファクタ
- 依存更新
- 脆弱性対応
- 調査

次のような場合は、機能チケットとは別に整理用チケットを切ります。

- 実装と同時に責務分割が必要になる場合
- 脆弱性対応が複数の更新経路に分かれる場合
- 調査だけを先行しないと実装方式を決められない場合

## BDD ルール

### 必須要件

- すべてのチケットに最低 1 本の `Given / When / Then` を書く
- 書き方は利用者や操作対象を含め、観測可能な結果で終える
- バグ修正は再現シナリオと修正後シナリオを持たせる
- リファクタや依存更新は「壊れてはいけない挙動」をシナリオ化する

### 記述例

```md
Given ユーザーが 32x32 の画像を読み込んでいる
When オート・アウトラインを実行する
Then 透過色の外周 1 ドットだけが現在色で塗られる
```

## チケットテンプレート

GitHub issue template では、以下の項目を標準テンプレートとします。

```md
## 背景

## ユーザーストーリー
As a ...
I want ...
So that ...

## BDDシナリオ
Given ...
When ...
Then ...

## 受け入れ条件
- ...

## テスト観点
- ...

## 実装メモ
- ...

## 非対象
- ...
```

## チケット種別ごとの運用

### 新機能

- ユーザーストーリーを必須とする
- UI の変化と操作結果を BDD で表現する
- 画面状態や出力物の期待結果を受け入れ条件に含める

### バグ修正

- 再現条件を背景と BDD に残す
- 修正前に何が壊れていたかを明記する
- 修正後に再発防止の確認観点を書く

### リファクタ

- 利用者から見える挙動を変えないことを前提にする
- 何の責務を分離するかを背景に書く
- 既存仕様維持を受け入れ条件に入れる

### 依存更新 / 脆弱性対応

- 更新対象と更新理由を背景に書く
- 影響範囲の確認内容をテスト観点に書く
- 修正経路ごとにチケットを分ける

## App.jsx の肥大化運用

現状の [src/App.jsx](/Users/mn/Documents/Codex/2026-04-26/github-plugin-github-openai-curated-https/dot-plate-pro/src/App.jsx) は 938 行です。

- 固定の行数閾値ではなく、責務増加を契機にチケット化する
- UI、状態管理、3D処理、入出力、編集ツールなどの責務が増える場合は、機能チケットとは別に分割チケットを切る
- 分割方針は「コンポーネント単位に分割していく」を基本にする
- 具体的な責務マップと優先順位は [App.jsx Responsibility Split Strategy](/Users/mn/Documents/Codex/2026-04-26/github-plugin-github-openai-curated-https/dot-plate-pro/docs/app-responsibility-split-strategy.md) を参照する

`Gemini.md` にはシングルファイル原則がありますが、これは現行制約として尊重しつつ、保守性課題はチケット化して解消対象にします。

## 脆弱性運用

脆弱性は依存 1件ごとではなく、**修正経路ごとにまとめてチケット化** します。

2026-04-26 時点で確認できている `npm audit` の結果は以下です。

- total: 8
- moderate: 3
- high: 5

最低限、次の単位でチケット化します。

- `vite / rollup` 系の更新と影響確認
- `postcss` 系の更新と影響確認

間接依存の `ajv`, `brace-expansion`, `flatted`, `minimatch`, `picomatch` は、対応元の更新チケットに含めて管理します。

## 運用フロー

1. GitHub issue を作成する
2. BDD シナリオと受け入れ条件を埋める
3. `feature/ticket-{id}-{slug}` ブランチを切る
4. 実装する
5. テスト観点を確認する
6. 1チケット 1PR でレビューに出す
